# RegEval experiment 2026-06-08-02 — challenger-v1

**Verdict:** DISCARD
**κ (final, post-challenger):** 0.380 (95% CI [0.225, 0.535])  vs champion 0.820
**κ (judge-only, Opus):** 0.770 (95% CI [0.658, 0.882]) — hidden finding, see below
**Diagnostics (final):** TPR=0.452 · TNR=0.925 · abst=0.080 · N=100
**Diagnostics (judge-only):** TPR=0.976 · TNR=0.975 · abst=0.080 · N=100
**Wall clock:** ~6:20 (32 agents, Opus model throughout)

**Hypothesis:** Adding an adversarial challenger sub-agent (Opus) that tries to REFUTE the judge's classification will improve κ beyond skills-v1 by catching over-committed calls. Per the Anthropic analytics blog: +6% accuracy at +32% token cost.

**Scaffold delta vs skills-v1:**
- Stage 1: identical judge prompt (skills-v1)
- Stage 2: adversarial challenger with `default to refuted: true if uncertain`
- Stage 3: resolution — refuted → flip compliant↔non-compliant; unclear stays unclear

**Why DISCARD:** Challenger catastrophically degraded κ by −0.390 from the judge-only baseline.
- Refutation rate: 37% (challenger refuted 37 items)
- flip_improved_count: **0** — not a single refutation improved the outcome
- flip_degraded_count: **24** — 24 flips made things worse
- TPR collapsed from 0.976 → 0.452: challenger was flipping correct compliant judgments to non-compliant at high rate

---

## Failure mode analysis

### Primary failure — "default refuted: true" + flip logic is wrong for this domain

The challenger prompt defaults to `refuted: true` when uncertain. At a 37% refutation rate, it refuted ~22 items that the judge had correct. With compliant items at 42% of the gold set, a trigger-happy refuter flips correct compliant → non-compliant at scale.

The flip resolution logic (compliant↔non-compliant) compounds the damage: every false refutation on a correct compliant item creates a false non-compliant output. There is no recovery mechanism — once flipped, the item is wrong.

**This is the domain mismatch:** The Anthropic analytics blog found adversarial challengers work when the judge tends to over-accept. In RegEval, the Opus judge is already precise (TPR=0.976, TNR=0.975) — there are almost no over-committed calls to catch. The challenger has nothing correct to refute and defaults to refuting anyway.

### Secondary finding (hidden positive) — Opus judge-only at κ=0.770

The Opus judge running skills-v1 prompt achieved **κ=0.770 with 86 correct, TPR=0.976, TNR=0.975** — without the challenger. Compared to Sonnet running the same prompt (κ=0.733, 83 correct, TPR=0.786):

| | Sonnet skills-v1 | Opus judge-only |
|---|---|---|
| κ | 0.733 | 0.770 |
| TPR | 0.786 | 0.976 |
| TNR | 0.975 | 0.975 |
| abst rate | 0.200 | 0.080 |
| Correct | 83 | 86 |

Opus naturally handles the description-vs-assertion distinction (skills-v1 failure mode 1) better than Sonnet, recovering 7 compliant items without the skills-v2 patch. The remaining 14 mismatches are in the unclear class and the ambiguous close-call items — the harder cases.

**Implication:** If skills-v2's Step 4 patch raises Sonnet to Opus-equivalent on the description items, the two paths should converge around κ=0.770–0.800. Whether Opus + skills-v2 can clear 0.820 KEEP bar is the next question.

## Challenger architecture verdict

The current flip resolution logic (compliant↔non-compliant) is not viable when:
1. The judge is already well-calibrated (κ>0.70)
2. The challenger defaults to refuting under uncertainty
3. The domain is imbalanced toward compliant items

**challenger-v2 options if revisiting:**
- Replace flip logic with "return to judge with challenger's counter-argument for a final verdict" (higher token cost, may recover)
- Raise challenger's refutation threshold: `refuted: true only if you are highly confident` (not just uncertain)
- Use challenger only on unclear items (current misses concentrated there) rather than all 100 items

**Recommendation:** Park challenger architecture. The model-upgrade path (Opus judge-only) and the skills-v2 patch are both higher-leverage.
